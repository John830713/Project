//==============================================================================
// Modules Layer - Design Overview
//==============================================================================

Purpose:
  Concrete feature modules, each is an independent functional unit.

Structure:
  Modules/<ModuleName>/
      <ModuleName>Module.h/.cpp   - Module body (implements IFeatureModule)
      <ModuleName>Window.h/.cpp   - Custom tool window (optional)
      <ModuleName>.module.ini     - Module metadata (required)
      <ModuleName>.rc             - Module resources (optional)

Module Metadata (<ModuleName>.module.ini):
  [Module]
  Name=ModuleName
  Sources=ModuleName.cpp
  Headers=ModuleName.h
  Resources=
  Libraries=
  Enabled=1

Rules:
   1. Every module must fully implement IFeatureModule.
   2. For drag-drop support, additionally implement IDropActionProvider.
    3. For right-click menu items, override GetContextMenuItems() / ExecuteContextMenuItem() from IFeatureModule (default: empty, auto-dispatched by ModuleManager; items are nested under the "Function" submenu).
       IMPORTANT: Labels returned by GetContextMenuItems() must be translated by the module
       itself using Tr(), e.g.: contextMenuItems[0].label = Tr(L"ModuleName", L"Toggle Enable").
       Raw English text will appear if Tr() is not called.
    4. For custom input, additionally implement IInputHandler.
    5. For About dialog, override GetVersion() / GetPurpose() from IFeatureModule (defaults: "1.0.0" / ""; shown in separator-separated list in About dialog).
    6. Modules communicate with host via IHostContext, NOT a concrete HostApp class.
    7. Settings accessed via GetValue/SetValue, persisted by ConfigManager.
    8. Config fields (GetConfigDefinitions) are automatically exposed in the Settings dialog; each module gets a tab. Bool→checkbox, Int/String→edit box.
    9. Build script auto-scans *.module.ini; no manual Makefile editing needed.

Translation conventions for modules:
  - Each module has its own lang file: Modules/<Name>/lang/zh-TW.ini
  - Build.py merges all modules' lang files into Translation/zh-TW.ini on each build.
  - Translation sections: [ModuleName] for general UI strings + context menu labels,
    [TabLabels] for settings tab display names, [About] for version/purpose strings.
  - Prefix+name pattern: When concatenating a translated prefix with a dynamic string,
    NEVER put a trailing space in the translation key or value (Build.py strips both).
    Instead, add the space in C++: Tr(L"AutoKey", L"▶ Start") + L" " + actionName.
  - About dialog module names: GetVersion()/GetPurpose() strings should be returned raw
    (not translated); the About dialog handles the Tr wrapping.
  - Context menu parent prefix: If a module's context menu items need a parent prefix
    (like "AutoKey ▶"), the prefix is translated by Tr() and the space between prefix
    and child label is added in C++.

Config file conventions:
  - Module config files live in Config/Config_<ModuleName>.ini.
  - Standard fields (GetConfigDefinitions) use [<ModuleName>] section.
  - Custom fields or multi-record data may use additional sections like [Action_N]
    (see AutoKey for an example with GetPrivateProfileStringW-based access).
